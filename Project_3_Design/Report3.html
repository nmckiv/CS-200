Coupling refers to what degree the modules of our system are interdependent on each other. We can look at our class diagrams to see how much of an impact these two concepts have. We don't see many bidirectional associations between classes which would otherwise suggest that classes have multiple instances to another. This would mean that changes in a class would affect changes in another class it is associated with. A system with high cohesion would have components that have methods and attributes that serve the purpose of the class. For example, in the “ProviderDirectory” entity class, we see the attributes and methods are closely related to the class responsibility that is to display the provider directory. A system with high cohesion and low coupling like ours, would make maintainability and modularity of the system easier over the lifespan of the system.

For the architecture of our project, we feel that a data Abstraction and Object-Oriented Organization style would be most suited. This architectural style aligns with the object oriented style of Java and it aligns with our use of abstraction as well. In our project, we have made sure to encapsulate a lot of the complex details of our project in classes so that the end-user does not have to deal with the complexities of the system. We have created classes that facilitate the use cases of our project such as the billing process for which the user(provider) only has to fill out a form and does not have to deal with the calculation of etf data, total amount payable and so on. In addition, the principles of inheritance and polymorphism allows us to simplify the development process as we are able to reuse code and maintain consistency such as how we have done with the report classes.
